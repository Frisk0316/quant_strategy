1. 系統總覽

這個 repo 不是單純策略 notebook，而是一套完整量化交易系統：資料、策略、投組、執行、風控、回測、API、前端與 AI 協作治理都有自己的模組。

核心設計理念

  • 所有交易邏輯透過 EventBus 串接，避免策略直接呼叫 execution。
  • 所有 order 進交易所前必須通過 RiskGuard。
  • PositionLedger 是 PnL 與持倉的 single source of truth。
  • 回測 replay 使用與 live 類似的 Strategy → Signal → Order → Fill → Ledger 路徑。
  • 前端 dashboard 由 FastAPI 靜態服務與 REST / WebSocket API 支援。

目前最重要的工程注意事項

  • PnL 高風險 SWAP PnL 必須乘上 ct_val。
  • 文件一致性 ADR 不能把目標架構寫成已實作行為。
  • Replay terminal liquidation / validation gates 需要測試確認。
  • Frontend ES module MIME type 必須是 JavaScript。
  • AI workflow Codex 只能依 issue scope 修改。

3. 事件流程：Live / Demo / Shadow 的主路徑

系統的核心是 event-driven flow。策略不直接下單；策略只產生 signal，PortfolioManager 轉 order，RiskGuard 檢查，ExecutionHandler 下單，Fill 再回寫 PositionLedger。

Market Data

OKX WebSocket / REST，books、trades、funding-rate。

EventBus

MARKET / FUNDING 事件分發。

Strategy

OBI / AS / Funding / Pairs 產生 SignalPayload。

Portfolio

根據 signal strength、vol target、risk multiplier 產生 OrderPayload。

RiskGuard

fat-finger、position limit、drawdown、daily loss、stale quote。

Execution

OrderManager + Broker，OKX / Sim / Shadow / Replay。

Fill / Ledger

Fill 回寫 PositionLedger，更新 PnL、equity、risk snapshot。

重要：錯誤排查時請先判斷是哪一層出錯：Data → Strategy → Portfolio → Risk → Execution → Ledger → API → Frontend。

7. Portfolio / PnL Accounting

PositionLedger 是持倉與 PnL 的 single source of truth。這一層錯了，策略績效與前端圖表都會失真。

Position

size 為 signed quantity：正數 long、負數 short。SWAP 的 size 是 contracts，不是 coin units。

ct_val 是合約乘數，例如 BTC-USDT-SWAP 常見為 0.01。

Unrealized PnL

SWAP:
size × (last_price - avg_entry) × ct_val

SPOT:
size × (last_price - avg_entry)

Equity

equity =
cash_equity
+ Σ position.unrealized_pnl

Fees 與 realized PnL 會更新 cash equity。

高風險錯誤：如果 SWAP PnL 忘記乘 ct_val，BTC-USDT-SWAP 可能出現約 100 倍 PnL / drawdown 錯誤。

10. Backtesting / Replay Layer

Replay backtest 不是單純 vectorized return，而是盡量走 Strategy → Signal → Portfolio → Order → Execution → Fill → Ledger 的事件路徑。

Replay Engine Flow

HistoricalEventFeed
  → MARKET / FUNDING events
  → EventBus
  → Strategy.on_market()
  → SIGNAL
  → PortfolioManager.on_signal()
  → ORDER
  → ExecutionHandler / ReplayExecutionModel
  → FILL
  → PositionLedger.on_fill()
  → ReplayRecorder
  → ReplayBacktestResult

輸出 artifacts

  • metrics.json：total_return、sharpe、max_drawdown、fill_rate 等。
  • orders.csv：所有 order event。
  • fills.csv：所有 fill event。
  • trades.csv：position ledger trade log。
  • equity_curve.csv：equity / drawdown / return。
  • funding.csv：funding settlement cashflow。
  • signals.csv、risk_events.csv、rejected_orders.csv。
注意：ADR-0005 中的 terminal liquidation / validation gates 目前應視為目標設計，需以 source code 與 regression tests 確認，不應直接假設已完整 enforce。

指標 截圖數值 代表內容 來源 / 公式 判讀方式
n_periods 735.00 回測 return series 的期間數量。 len(returns) 用於年化 Sharpe / Sortino / Calmar；bar 不同時不可直接比較。
total_return -0.1070 整段期間的累積總報酬。 prod(1 + r) - 1 -0.1070 = -10.70%。但若途中 bankrupt，最後回來也不代表可交易。
sharpe -3.60 年化平均超額報酬 / 波動率。 sqrt(periods) * mean(r-rf) / std(r-rf) 越高越好；負值表示風險調整後報酬為負。
sortino -2.55 只用 downside deviation 的風險調整報酬。 sqrt(periods) * mean(excess) / downside_std 比 Sharpe 更聚焦下檔風險；負值同樣不好。
max_drawdown -1.38 最大回撤比例。 equity cumulative max 對比當前 equity。 -1.38 = -138%，已超過 100%，代表 equity 曾跌破零。
calmar -0.5347 年化 CAGR / 最大回撤絕對值。 CAGR / abs(max_drawdown) 越高越好；負值表示 CAGR 為負或路徑極差。

14. 錯誤處理與 Debug Runbook

系統出錯時，不應直接要求 AI「幫我修」。應先收集 evidence、定位 layer、建立 issue，再讓 Claude plan / Codex implement。

Frontend Blank Page

  1. 打開 DevTools Console。
  2. 檢查 module import / MIME error。
  3. curl -I /app.js。
  4. 檢查 /api/backtest/runs 是否 500。
  5. 確認 index.html script src。

Replay Result Weird

  1. 看 fills.csv。
  2. 看 trades.csv。
  3. 確認 SWAP ct_val。
  4. 確認 funding sign。
  5. 檢查 open positions at end。

Pairs Hedge Issue

  1. entry metadata 是否有 hedge。
  2. exit / stop 是否同時關 hedge。
  3. PositionLedger 是否有 orphan position。
  4. 若測到 bug，獨立開 fix PR。
AI debug 原則：先 bug report，再 plan，再 implementation。不要把整個錯誤直接丟給 Codex 自由修。
