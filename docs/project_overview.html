OHLCV 不變量（Invariants）

規則條件違反時
高價 ≥ 所有四價high ≥ open AND high ≥ close AND high ≥ low標記為 suspect，記錄品質事件
低價 ≤ 所有四價low ≤ open AND low ≤ close AND low ≤ high標記為 suspect
所有價格 > 0open, high, low, close > 0拒絕寫入
異常值偵測|z_score| > 3.0（與滾動均值比較）標記 outlier_flagged，sigma_threshold 設定於 settings.yaml

ADR-0005：Replay 驗證四關卡

關卡名稱判斷條件失敗處理
Gate 1 終端倉位清算 回測結束時（liquidate_on_end=True），所有倉位以最後中間價平倉後，PositionLedger 應顯示零倉 設定 bankrupt=True
Gate 2 成交率警告 fill_rate < 5% 且 submitted_order_count > 0 記錄 Warning，檢查 order_latency_ms 設定
Gate 3 最低資料覆蓋率 每個標的覆蓋率 ≥ 80% Replay 開始前拋出 ValueError
Gate 4 資金費率覆蓋 funding_carry 回測中 funding_settlement_count > 0 記錄 Warning（策略可能未收到資金費）

§3回測系統邏輯與參數設定

執行時間線（無前視偏差）

close[t]
收盤 K 線
→
signal[t]
計算特徵與排名
→
weight[t]
目標權重（調倉頻率）
→
ffill → 1m
前向填充至分鐘
→
shift(1)
延遲一棒執行
→
position[t]
實際持倉
→
return[t] = position[t] × bar_return[t]
關鍵：rolling_high 使用 high.shift(1)，確保 t 時刻的突破判斷不包含當棒最高價，徹底避免前視偏差。

回測完整流程

  1. 載入資料DataLoader 從 parquet / PostgreSQL 讀取各標的 1m OHLCV，合併為寬格式 DataFrame（index = UTC timestamp，columns = inst_id）
  2. 特徵工程計算 return_fast、return_slow、volume_z、realized_vol、rolling_high、EMA、ATR，全部使用 lagged 資料
  3. 橫截面評分對每個調倉時間點做 row-wise z-score，加權混合為複合分數
  4. 趨勢過濾判斷 benchmark（BTC）是否在牛市（close > EMA）；不在則全部歸零
  5. 進場過濾排名前 top_k + return_fast > 0 + return_slow > 0 + 突破過濾（close > rolling_high AND close > EMA）
  6. 出場規則（有狀態）逐棒掃描持倉，觸發任一出場條件即平倉（趨勢翻空 / 排名跌出緩衝 / EMA 跌破 / ATR 停損 / 超時）
  7. 上採樣 + 延遲執行rebalance 頻率的權重 ffill 至 1m，shift(1) 確保下一棒才執行
  8. 計算報酬與成本gross = position × bar_return；cost = turnover × (fee_bps + slippage_bps) / 10,000（只在調倉棒扣除）
  9. 輸出結果equity_curve、daily_returns、positions、target_weights、trades、40+ metrics
